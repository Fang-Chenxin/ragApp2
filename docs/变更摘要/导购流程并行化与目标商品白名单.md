# 导购流程并行化与目标商品白名单 - 变更说明

## 变更时间
2026-06-03

## 变更概述

本次变更将导购聊天流程从单一串行链路调整为多分支并行执行，并拆分需求分析、商品查询规划、最终导购回复三类提示词职责。最终推荐不再直接依赖工具规划轮的自由回复，而是由后端先生成结构化目标商品白名单，再交给导购推荐 LLM 基于白名单、RAG 补充信息和需求分析生成最终回复。

同时，本次提交包含 RAG 商品知识库接入增强、RAG 来源事件透传、真实向量库测试脚本、默认数据路径修正和 LLM 温度参数处理修复，确保测试流程可以逐步落地到实际后端流式接口。

---

## 后端变更

### 1. 工具聊天服务 (`backend/service/tool_chat_service.py`)

- **修改位置**: `chat_with_tools_stream()`、`_build_need_analysis_messages()`、`_build_system_prompt()` 等方法
- **新增特性**:
  - 支持从向量检索结果中提取结构化 RAG 来源，并通过 `rag_sources` 事件返回。
  - 增加 RAG 向量检索超时和 LLM RAG 核验，避免 Chroma 或核验请求长时间阻塞聊天。
  - 需求分析 LLM 不再依赖 RAG 上下文，只基于用户问题和历史对话生成内部摘要。
  - 新增 `_build_tool_planning_prompt()`，工具查询规划只负责调用 `query_products`，不读取知识库上下文。
  - 最终导购 prompt 只在汇总阶段使用 RAG 上下文，并明确商品事实以工具结果和目标商品白名单为准。
  - 流式流程提前并行启动需求分析、第一轮工具规划和原始需求 SQLite 直查，减少串行等待。
  - 新增 `_build_target_products()`，将直查商品和工具查询商品合并为结构化目标商品列表。
  - 当存在目标商品时，最终回复强制走受约束导购生成，不再直接放行工具规划轮中的 LLM 回复。

- **目标商品白名单字段**:

```json
{
  "rank": 1,
  "product_id": "p_beauty_025",
  "title": "花西子螺黛生花眉笔",
  "brand": "花西子",
  "category": "美妆护肤",
  "sub_category": "眉笔",
  "base_price": 89.0,
  "source": "direct_query",
  "recommendation_role": "primary"
}
```

- **向后兼容**: 基本兼容。`selected_products` 事件仍然返回 `selected_product_ids` 和 `selected_products`，但 `selected_products` 中新增 `rank`、`source`、`recommendation_role` 等字段。

### 2. 后端 API 层 (`backend/api/chat.py`)

- **修改内容**:
  - 流式接口新增 `rag_sources` 事件转发，客户端可获取知识库来源商品和片段信息。
  - 后端耗时日志新增 `RAG检查` 字段，和服务层 `rag_rerank` 计时保持一致。

### 3. RAG 服务与配置

#### 3.1 RAG 服务 (`backend/service/rag_service.py`)

- **修改内容**:
  - 新增 `query_with_sources()`，返回文档内容、向量距离和商品 metadata。
  - 新增 `format_results_as_context()`，统一将带来源的知识片段格式化为 prompt 上下文。
  - 保留 `query()` 旧接口，仅返回纯文本，保证旧调用兼容。

#### 3.2 配置 (`backend/config/settings.py`)

- **修改内容**:
  - 默认 Chroma 路径切换到项目内 `ecommerce_agent_dataset/.chroma`。
  - 默认集合名切换为 `product_knowledge`。
  - 默认 SQLite 商品库路径切换到项目内 `ecommerce_agent_dataset/ecommerce.db`。
  - 新增 RAG 向量检索超时、LLM 核验开关、核验超时和最低保留分配置。

#### 3.3 LLM 服务 (`backend/service/llm_service.py`)

- **修改内容**:
  - 修复 `temperature=0.0` 被 `or` 表达式误替换为默认温度的问题。
  - 现在仅当 `temperature is None` 时才使用 `settings.rag_temperature`。

### 4. 真实流程测试追踪 (`test/test_real_tool_chat_flow.py`)

- **修改内容**:
  - 将测试追踪阶段标题从顺序编号改为并行分支展示。
  - 补充后台直查耗时、目标商品来源和推荐角色打印。
  - 耗时汇总新增“各阶段串行相加”和“并行重叠节省估算”，避免并行后继续误读为各阶段相加等于总耗时。

### 5. RAG 向量查询测试 (`test/test_rag_vector_query.py`)

- **新增内容**:
  - 验证默认 Chroma 配置是否指向电商商品知识库。
  - 验证 `query_with_sources()` 能返回眉笔商品知识片段和 metadata。
  - 验证 `format_results_as_context()` 能输出带商品 ID 和片段类型的上下文。

### 6. 流式回归测试 (`test/test_tool_chat_service_streaming.py`)

- **修改内容**:
  - 更新正常工具对话流式测试的 LLM 调用计数预期。
  - 新流程下存在目标商品时会额外进入最终受约束生成，因此 `chat_stream` 调用次数从 1 次变为 2 次。

## 当前流程说明

### 并行分支

```text
进入 chat_with_tools_stream
  -> 并行启动：
      A. 需求分析 LLM
      B. 第一轮工具规划 LLM
      C. 原始需求 SQLite 直查
  -> 主流程继续：
      D. 向量检索
      E. RAG 核验
```

### 汇总与最终生成

```text
工具查询结果 + 原始直查结果
  -> 后端生成目标商品白名单
  -> 最终导购 LLM 读取：
      1. 目标商品白名单 JSON
      2. 工具查询结果
      3. RAG 核验后的知识库上下文
      4. 需求分析摘要
  -> 生成最终导购回复
```

## 风险与后续落地计划

| 风险项 | 描述 | 缓解措施 |
|------|------|---------|
| LLM 并发限制 | 部分 OpenAI-compatible 服务可能限制同一 key 并发请求 | 实际应用接入时增加并发开关或降级为串行执行 |
| 目标商品排序策略较简单 | 当前按直查优先、工具查询补充排序，尚未引入评分融合 | 后续可加入工具结果得分、RAG 来源关联度和业务规则权重 |
| 前端展示仍只消费部分字段 | Android 端当前主要展示回复文本和耗时，未显式展示目标商品卡片 | 后续 API/客户端接入 `selected_products` 的结构化字段 |

### 实际应用落地计划

1. **后端接口确认**
   - 确认 `/api/chat/stream` 当前已消费 `chat_with_tools_stream()` 的 `selected_products`、`content`、`done` 事件。
   - 将 `selected_products` 中新增的 `rank`、`source`、`recommendation_role` 字段纳入接口契约说明。
   - 增加并发开关配置，例如 `TOOL_CHAT_PARALLEL_ENABLED=true/false`，用于不支持并发 LLM 的环境回退。

2. **客户端展示接入**
   - Android 端继续展示最终回复文本，同时保留 `selected_products` 结构化数据。
   - 后续可新增目标商品卡片区域，按 `rank` 展示商品标题、品牌、价格和来源。
   - 客户端不从自然语言回复中解析商品 ID，而以 `selected_products` 为外部系统对接依据。

3. **推荐真实性校验**
   - 后端在 `done` 前校验最终回复中出现的 `p_*_*` 商品 ID 是否属于目标商品白名单。
   - 如 LLM 输出白名单外 ID，触发确定性兜底或重新生成。
   - 增加测试用例覆盖“RAG 未命中但工具命中”“LLM 输出错误 ID”“直查与工具重复商品去重”。

4. **性能指标验证**
   - 对比并行开关开启/关闭时的 `total`、`analysis_calls`、`rag_rerank`、`llm_calls`。
   - 在调试输出中继续保留“各阶段串行相加”和“并行重叠节省估算”。
   - 真实环境下重点观察第一轮工具规划是否与 RAG/需求分析有效重叠。

5. **排序策略迭代**
   - 当前版本采用直查优先、工具查询补充的保守策略。
   - 后续可引入 RAG 来源关联、工具查询排序、价格/品类规则和用户历史偏好形成综合排序。

## 用户使用说明

1. 用户继续正常发送导购问题。
2. 后端会并行执行需求分析、商品查询规划、SQLite 直查和 RAG 检索核验。
3. 最终推荐会基于后端生成的目标商品白名单，避免编造商品 ID 或引用外部不可核对商品。
4. 调试测试中可查看每个目标商品的 `rank`、`source` 和 `recommendation_role`。
