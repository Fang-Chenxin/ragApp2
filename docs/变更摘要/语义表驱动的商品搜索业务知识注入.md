# 语义表驱动的商品搜索业务知识注入 - 变更说明

## 变更时间
2026-06-08

## 变更概述
本次变更为 LLM SearchPlan 引入了结构化的业务知识语义表，实现「LLM 负责语义规划，扩表负责给 LLM 和程序提供稳定业务边界，代码只做执行和校验」的核心原则：

- 新增 6 个 JSON 语义表文件，覆盖商品概念、品类别名、fallback 关系、品牌型号、场景标签和回归测试用例
- 新增语义表加载与查询服务，提供全局单例访问
- 改造 SearchPlan prompt 注入，将语义表命中结果注入 LLM system prompt
- 改造商品匹配逻辑，用语义表规则做确定性修正，替代代码中的零散词表
- 新增回归测试脚本，支持模拟模式和真实 SQLite 全链路模式

---

## 后端变更

### 1. 语义表数据文件 (`ecommerce_agent_dataset/search_semantics/`)

新增 6 个 JSON 数据文件，作为 SearchPlan 的业务知识输入源：

| 文件 | 条目数 | 用途 |
|------|--------|------|
| `product_concepts.json` | 30 | 商品概念同义表：口语词映射到 direct_terms / fallback_terms |
| `category_aliases.json` | 28 | 品类/子品类别名：用户词映射到库内真实 category / sub_category |
| `fallback_relations.json` | 10 | Fallback 关系：定义可接受替代和禁止类目 |
| `brand_model_aliases.json` | 10 | 品牌型号表：strict_direct 约束（iPad / MacBook 等） |
| `scenario_tags.json` | 12 | 场景标签：学习 / 游戏 / 敏感肌等场景的检索提示 |
| `regression_cases.json` | 15 | 回归测试用例：覆盖四大品类 + 无库存场景 |

### 2. 语义表服务 (`backend/service/product_search/search_semantics_service.py`)

- **新增文件**: 全局单例 `search_semantics_service`
- **核心能力**:
  - 加载并缓存 6 个 JSON 语义表
  - `match_product_concepts(query)` — 从 query 匹配商品概念
  - `match_category_aliases(query)` — 从 query 匹配品类别名
  - `resolve_category(term)` — 将用户词解析到库内真实 category/sub_category
  - `get_fallback_relation(query)` — 获取 fallback 可接受/禁止规则
  - `match_brand_models(query)` — 匹配品牌型号约束
  - `match_scenario_tags(query)` — 匹配场景标签
  - `build_search_plan_hints(query)` — 构建注入 prompt 的业务知识提示文本
  - `get_effective_forbidden_categories()` / `get_effective_allowed_categories()` — 合并语义表与 SearchPlan 的品类约束

### 3. SearchPlan Prompt 注入 (`backend/service/tool_chat/prompts.py`)

- **修改方法**: `_build_search_plan_messages()` (L80+)
- **变更内容**:
  - 调用 `search_semantics_service.build_search_plan_hints(user_query)` 生成业务知识提示
  - 将提示注入 system prompt，新增规则 5：「如果下方业务知识参考中有 direct_terms、fallback_terms 等信息，必须优先使用」
  - 提示包含：商品概念知识、品类真实值、Fallback 规则、品牌型号约束、场景标签

### 4. SearchPlan 确定性修正 (`backend/service/tool_chat/product_selection.py`)

- **新增方法**: `_apply_semantic_corrections(plan, user_query)` (L57+)
- **修正逻辑**:
  1. 品类别名修正：LLM 留空时用 `resolve_category()` 补全 target_category/target_sub_category
  2. 品牌型号修正：strict_direct 品牌的 direct_terms 强制注入
  3. 商品概念修正：语义表命中的 direct_terms 注入，替代含意图词的原始 query
  4. Fallback 关系修正：注入 forbidden_categories
  5. 品类约束推断：allowed_categories=1 时自动推断 forbidden_categories（其余 3 个）

- **修改方法**: `_is_direct_product_match()` (L310+)
  - 从 `brand_model_aliases` 动态扩展 `specific_model_tokens`
  - strict_direct 模式下必须标题/品牌含该词

- **修改方法**: `_matches_user_product_constraints()` (L380+)
  - 新增语义表 `forbidden_categories` 兜底过滤
  - `semantic_terms` 检查改为仅在无 `allowed_categories` 时生效，避免 fallback 结果被误杀

- **修改方法**: `_prefer_closest_fallbacks()` (L450+)
  - 新增 `fallback_relations` 的 acceptable 规则优先排序

## 测试变更

### 5. 数据校验脚本 (`test/validate_search_semantics.py`)

- **新增文件**: 校验 6 个 JSON 文件的结构完整性
- **校验项**: 类型检查、必需字段、类目合法性、ID 唯一性等

### 6. 回归测试脚本 (`test/test_search_plan_regression.py`)

- **新增文件**: 基于 `regression_cases.json` 的自动化回归测试
- **两种模式**:
  - 模拟模式（默认）：用 17 个假商品测试语义表匹配 + 品类过滤
  - `--live` 模式：走真实 SQLite 查询 + `_build_target_products` 全链路
- **验证项**: match_type、allowed_product_ids、forbidden_product_ids、allowed_categories、forbidden_categories
- **15 条用例覆盖**: iPad/MacBook/口红/防晒/跑步鞋/瑜伽裤/咖啡/酸奶/肉松面包/猫粮(无库存)/敏感肌/T恤/篮球鞋/蓝牙耳机

## 核心问题修复清单

| 问题描述 | 修复状态 |
|---|---|
| LLM SearchPlan 品类值靠猜测，经常与库内值不一致 | ✅ 已修复（category_aliases 注入 prompt + 确定性修正） |
| 代码中零散词表（beauty_terms/digital_terms 等）难以维护 | ✅ 已修复（语义表驱动替代硬编码） |
| 品牌型号（iPad/MacBook）被普通平板/笔记本混入 direct 结果 | ✅ 已修复（brand_model_aliases strict_direct 约束） |
| 无库存品类（猫粮）没有禁止类目约束 | ✅ 已修复（fallback_relations forbidden_categories） |
| 缺少回归测试，改一处容易坏另一处 | ✅ 已修复（regression_cases.json + 测试脚本） |
| 用户口语词（"游戏本"、"防晒霜"）无法映射到检索词 | ✅ 已修复（product_concepts direct_terms 注入） |

## 用户使用说明

1. **扩表流程**: 发现推荐跑偏时，先补 `regression_cases.json`，再按需补对应的语义表文件
2. **校验数据**: `python test/validate_search_semantics.py`
3. **运行回归测试**: `python test/test_search_plan_regression.py`（模拟）或 `python test/test_search_plan_regression.py --live -v`（真实）
4. **不再往代码里加零散词表**: 所有业务知识通过语义表管理
