# 导购购买意图识别与回复口径优化 - 变更说明

## 变更时间
2026-06-09

## 变更概述

本次变更在后端导购链路中新增“随便看看 vs 明确购买倾向”的轻量结构化识别能力。系统会在 SearchPlan 中输出 `purchase_intent` 和 `purchase_intent_reason`，并在最终推荐阶段根据购买阶段调整回复口径：探索型用户仍获得商品推荐，但语气更偏参考和对比；明确购买型用户继续获得清晰的优先级推荐和取舍建议。

---

## 后端变更

### 1. SearchPlan Prompt (`backend/service/tool_chat/prompts.py`)

- **修改位置**: `_build_search_plan_messages()`
- **新增特性**:
  - SearchPlan JSON schema 新增 `purchase_intent` 与 `purchase_intent_reason`。
  - Prompt 明确要求识别 `browsing` 和 `purchase_ready` 两类购买阶段。
  - “随便看看、看看有没有、了解一下、有什么可选、先看看”归入 `browsing`。
  - “想买、我要、需要、下单、预算、给我推荐一款、急用”归入 `purchase_ready`。
- **向后兼容**: 是，旧模型未输出新字段时会进入规则兜底。

### 2. SearchPlan 标准化 (`backend/service/tool_chat/product_selection.py`)

- **修改位置**: `_normalize_search_plan()`、新增 `_normalize_purchase_intent()` 与 `_infer_purchase_intent_from_query()`
- **新增特性**:
  - 将购买阶段写入标准化 SearchPlan，供后续阶段统一消费。
  - 模型返回合法字段时直接保留。
  - 模型未返回字段时，根据用户原话确定性推断。
  - 模型返回非法值时按 `purchase_ready` 兜底，避免弱化正常导购推荐。
- **向后兼容**: 是，不改变商品召回数量和目标商品白名单策略。

### 3. 最终推荐口径 (`backend/service/tool_chat/product_selection.py`, `backend/service/tool_chat/stream_final.py`)

- **修改位置**: `_build_final_recommendation_messages()` 与最终回复阶段调用
- **新增特性**:
  - `browsing`：最终 prompt 要求“轻推荐、不催买”，强调“可以先看这几个方向”“先帮你筛几款参考”、适用场景与差异点。
  - `purchase_ready`：保持原有明确导购风格，继续输出推荐优先级、适配理由和取舍建议。
  - `ctx.search_plan` 透传到最终回复构造函数，保证回复口径与 SearchPlan 识别结果一致。

### 4. 调试追踪 (`backend/service/tool_chat/stream_stages.py`, `backend/service/tool_chat/stream_final.py`, `backend/service/tool_chat/trace.py`)

- **修改位置**: SearchPlan debug、selected_products debug、可读日志格式化
- **新增特性**:
  - debug payload 增加 `purchase_intent` 和 `purchase_intent_reason`。
  - `backend.log` 的“LLM 商品搜索计划”和“最终目标商品合并结果”段落显示购买意图和判断依据。
- **收益**:
  - 应用内调用后可直接在日志中验证意图识别是否生效。

## 意图判断如何影响处理过程

| 阶段 | `browsing` 影响 | `purchase_ready` 影响 |
| --- | --- | --- |
| SearchPlan | 记录为探索/浏览阶段，并写入判断依据 | 记录为明确导购/购买阶段，并写入判断依据 |
| 商品召回 | 不减少召回数量，仍按现有 RAG、SQLite 直查、工具查询流程执行 | 保持原有召回策略 |
| 目标商品白名单 | 不改变 3-5 个候选合并和 direct/fallback 过滤策略 | 保持原有白名单策略 |
| 最终回复 | 语气更轻，强调参考、方向、对比，不使用强成交话术 | 输出明确推荐优先级、适配理由和取舍建议 |
| 调试日志 | 日志显示 `购买意图: browsing` 和判断依据 | 日志显示 `购买意图: purchase_ready` 和判断依据 |

## 测试变更

### 1. 单元测试 (`test/test_history_and_stream_safety.py`)

- **新增测试类**: `PurchaseIntentTest`
- **覆盖内容**:
  - “随便看看有没有适合跑步的耳机”识别为 `browsing`。
  - “我想买个跑步耳机”“推荐一款防晒霜”识别为 `purchase_ready`。
  - 非法 `purchase_intent` 值兜底为 `purchase_ready`。
  - `browsing` 会进入轻推荐、不催买的最终 prompt 约束。

### 2. SearchPlan 回归 (`test/test_search_plan_regression.py`)

- **新增用例**:
  - `intent_browsing_running_earphones`
  - `intent_purchase_running_earphones`
  - `intent_purchase_sunscreen`
- **验证点**:
  - SearchPlan 回归输出中包含预期的 `purchase_intent`。
  - 现有 22 条模拟回归用例全部通过。

## 用户使用说明

1. 用户输入“随便看看有没有适合跑步的耳机”时，系统会正常推荐商品，但回复更偏“先看方向、对比参考”。
2. 用户输入“推荐一款防晒霜”或“我想买个跑步耳机”时，系统按明确购买倾向输出更直接的推荐建议。
3. 开发调试时可在 `backend/backend.log` 的 SearchPlan 或 selected_products 段落查看“购买意图”和“判断依据”。
