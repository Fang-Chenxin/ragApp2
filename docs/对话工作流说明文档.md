# 电商导购对话工作流说明文档

## 概述

本系统是一个基于 RAG（检索增强生成）+ 工具调用的电商导购对话服务。用户通过自然语言描述购物需求，系统经过**需求分析、知识库检索、商品搜索计划、多轮工具调用、目标商品校验、最终流式回复**六大阶段，生成高质量的商品推荐回复。

## 整体架构

```
用户输入
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│              Stream Pipeline（流式编排引擎）              │
│                                                         │
│  ① parallel_bootstrap  ── 并行启动：需求分析 + 搜索计划 + 首轮LLM规划
│  ② rag_pipeline        ── 向量检索 + LLM Rerank 校验
│  ③ finish_analysis     ── 等待分析完成，构建最终 prompt
│  ④ direct_product_query ── 按 SearchPlan 直查 SQLite 商品库
│  ⑤ tool_loop           ── 多轮 LLM 工具规划 → SQLite 查询 → 结果回填
│  ⑥ final_reply         ── 目标商品合并校验 → 约束式流式回复
│                                                         │
└─────────────────────────────────────────────────────────┘
  │
  ▼
SSE 事件流 → 客户端
```

## 详细阶段说明

### 阶段 ①：并行启动（parallel_bootstrap）

**入口**：`_stream_stage_parallel_bootstrap`

**职责**：同时启动三个互不依赖的后台任务，缩短总耗时。RAG 向量检索和 SQLite 直查不在本阶段直接完成：RAG 在下一阶段启动；直查在 SearchPlan 结果可用后执行。

| 后台任务 | 说明 |
|---------|------|
| `analysis_task` | 需求分析 —— LLM 流式输出 1~3 句需求理解摘要 |
| `search_plan_task` | 商品搜索计划 —— LLM 输出结构化 JSON，含目标商品词、直接命中词、兜底词、允许/禁止类目 |
| `first_tool_planning_task`（并行模式） | 首轮工具规划 —— LLM function calling 的第一次调用，提前开始 |

**前端可见事件**：
- `status`: 正在分析需求
- `status`: 正在检索知识库

**日志标识**：
```
═══ [chat_with_tools_stream] 开始处理请求
  用户问题: xxx
  历史消息数: N
  最大工具调用轮数: 5
  使用模型: xxx
  并行流程: 启用
```

---

### 阶段 ②：RAG 管线（rag_pipeline）

**入口**：`_stream_stage_rag_pipeline`

**职责**：从 ChromaDB 向量库检索相关知识片段，并通过 LLM rerank 筛选高质量结果。

**子流程**：

1. **向量检索**（`_query_context_docs_with_timeout`）
   - 在线程池中执行，带超时保护（默认 3s）
   - 按 `user_query` + 可选 `category_filter` 检索 top-3 知识片段
   - 返回带 metadata 的结构化片段（product_id、title、category、distance、content）

2. **LLM Rerank 校验**（`_rerank_context_docs_with_llm`）
   - 构造评分 prompt，要求 LLM 对每个候选片段打 1~5 分
   - score ≥ 2 为保留阈值；低于阈值的片段被丢弃
   - 超时（默认 >30s）时保留原始向量排序
   - 解析失败时也保留原始排序

3. **穿插输出需求分析增量**
   - 向量检索期间，前端不空等，优先输出已生成的分析片段

**前端可见事件**：
- `status`: 正在校验知识库
- `analysis`: 需求分析增量（流式）
- `rag_sources`: 已定位知识来源商品

**日志标识**：
```
[RAGVectorScore] query=xxx | category_filter=xxx | rank=1 | id=xxx | title=xxx | distance=xxx
[RAGRerankScore] query=xxx | requested_id=xxx | score=5 | kept=True | reason=xxx
LLM RAG 检查未达到保留阈值，已丢弃知识库上下文
```

---

### 阶段 ③：完成分析（finish_analysis）

**入口**：`_stream_stage_finish_analysis`

**职责**：
1. 等待需求分析任务完成，排空分析队列
2. 构建最终回复阶段使用的 system prompt（注入 RAG 上下文 + 历史对话）

**日志标识**：
```
分析耗时: 5.145s
分析摘要: 用户的核心需求是...
分析完整内容: ...
构建消息列表: 2 条 (含 system + 历史 + 当前问题)
```

### 阶段 ④：商品直查（direct_product_query）

**入口**：`_stream_stage_direct_product_query`

**职责**：按 SearchPlan 的 query_text 直接查询 SQLite 商品库，提供兜底候选商品。

**子流程**：

1. **收集 SearchPlan 结果**
   - 如果在并行阶段已生成，直接取回
   - 否则同步生成

2. **LLM 商品搜索计划**（`_stream_run_search_plan`）
   - LLM 输出结构化 JSON：`target_product`、`query_text`、`direct_terms`、`acceptable_fallback_terms`、`allowed_categories`、`forbidden_categories`
   - 语义表修正（`_apply_semantic_corrections`）：
     - 品类别名修正
     - 品牌型号修正
     - 商品概念修正
     - Fallback 关系修正
     - 品类约束修正

3. **SQLite 直查**（`_query_direct_selected_products`）
   - 按 SearchPlan 的 query_text + direct_terms 组合查询
   - 返回最多 5 个候选商品

**前端可见事件**：
- `trace`: LLM 商品搜索计划
- `trace`: 原始需求召回 SQLite 商品库

**日志标识**：
```

  目标商品: xxx
  主检索词: xxx
  直接命中词: [...]

```

### 阶段 ⑤：工具调用循环（tool_loop）

**入口**：`_stream_stage_tool_loop`

**职责**：多轮"LLM 规划 → SQLite 工具执行 → 工具结果回填"循环，最多 5 轮。

**每轮循环流程**：

```
LLM 调用（function calling）
  │
  ├─ 有 tool_calls → 解析参数 → 并发执行 SQLite 查询 → 结果回填 messages → 下一轮
  │
  └─ 无 tool_calls → 进入直接回复整理（direct_reply_finalization）
```

**关键细节**：

1. **首轮复用并行结果**
   - 如果在阶段①已并行启动首轮 LLM 规划，这里直接等待结果
   - 计算并行节省的预估时间

2. **工具执行**
   - 同一轮多个 tool_calls 并发执行（`asyncio.as_completed`）
   - 每次调用记录：tool_call_id、tool_name、arguments、result、ok、total、elapsed

3. **提前退出条件**
   - LLM 无 tool_calls → 直接回复
   - 已获得足够目标商品 → 提前进入最终推荐
   - LLM 调用异常 → 输出错误并终止

4. **目标商品过滤**（`_build_target_products`）
   - 合并原始召回 + 工具召回候选
   - 回查 SQLite 数据库校验
   - 应用品类约束（allowed/forbidden categories）
   - 标记 match_type: `direct`（直接匹配）或 `fallback`（兜底替代）

**前端可见事件**：
- `trace`: LLM 第 N 轮工具调用计划
- `status`: 正在查询商品
- `trace`: 工具结果

**日志标识**：
```
  ── LLM 第 1 轮调用 ──
      发送消息数: 2
      LLM 响应完成 | 耗时: 2.898s | prompt_tokens=1068, completion_tokens=258
```

### 阶段 ⑥：最终回复（final_reply）

**入口**：`_stream_stage_final_reply` 或 `_stream_stage_direct_reply_finalization`

**职责**：合并目标商品，生成受约束的流式推荐回复。

**子流程**：

1. **目标商品合并**（`_stream_selected_product_payloads`）
   - 合并原始召回 + 工具召回
   - 回查数据库校验
   - 应用品类约束过滤
   - 输出最终目标商品白名单（最多 5 个）

2. **最终回复生成**
   - 两种模式：
     - `constrained_direct`：LLM 在工具循环中直接给出回复，但存在目标商品 → 再生成一次受约束回复
     - `final`：工具循环耗尽，执行最终流式 LLM 调用
   - 构建 final_messages：system prompt（含 RAG 上下文 + 目标商品清单）+ 需求分析 + 历史对话 + 用户问题
   - 流式生成，逐 chunk 输出给前端
   - 回复内容清洗（`_sanitize_user_reply_chunk`）：移除内部标记、商品 ID 等

3. **兜底机制**
   - 如果 LLM 流式返回空内容 → 使用确定性模板回复
   - 如果无目标商品 → LLM 自由回复（告知用户无匹配商品）

4. **耗时汇总与保存**
   - 输出各环节耗时
   - 保存思考过程（ThinkingSave）

**前端可见事件**：
- `trace`: 最终目标商品合并结果
- `selected_products`: 已选定目标商品
- `status`: 正在整理结果
- `trace`: 最终回复整理检查 / 最终整理输入
- `content`: 流式回复内容（逐 chunk）
- `done`: 完成事件（含 timings）

**日志标识**：

## 核心数据流

### SearchPlan 结构

```json
{
  "target_product": "适合新手的画眉笔",
  "target_category": "美妆护肤",
  "target_sub_category": "眉笔",
  "query_text": "画眉笔",
  "direct_terms": ["画眉笔", "眉笔"],
  "acceptable_fallback_terms": ["眉粉", "眉膏", "眉妆", "眉形工具"],
  "allowed_categories": ["美妆护肤"],
  "forbidden_categories": ["食品饮料", "数码电子", "服饰运动"],
  "fallback_notice_required": true
}
```

### 目标商品结构

```json
{
  "rank": 1,
  "product_id": "p_beauty_025",
  "title": "花西子螺黛生花眉笔细节点描顺滑显色防晕染持久双头眉笔0.08g",
  "brand": "花西子",
  "category": "美妆护肤",
  "sub_category": "眉笔",
  "base_price": 89.0,
  "source": "tool_query",
  "recommendation_role": "primary",
  "match_type": "direct",
  "match_note": "直接匹配用户要找的商品关键词"
}
```

### SSE 事件类型

| type | 说明 | content |
|------|------|---------|
| `status` | 状态变更 | "正在分析需求"、"正在检索知识库"、"正在查询商品"、"正在整理结果" |
| `analysis` | 需求分析增量 | 流式文本片段 |
| `rag_sources` | 知识来源商品 | 已定位的知识来源商品列表 |
| `selected_products` | 目标商品 | 已选定的目标商品 ID 列表 |
| `content` | 回复内容 | 流式回复文本片段 |
| `done` | 完成 | 空内容 + timings |
| `error` | 错误 | 错误描述 + timings |
| `trace` | 调试信息 | 各阶段详细中间结果（仅 debug 模式） |

## 典型场景示例

### 场景 1：精确匹配（"适合新手的画眉笔"）

1. 向量检索命中花西子眉笔的 review/marketing/faq 片段（distance=0.25）
2. LLM Rerank 全部保留（score=5）
3. SearchPlan：target=画眉笔，direct_terms=[画眉笔]，allowed=[美妆护肤]
4. 工具调用 1 轮即命中 p_beauty_025
5. 目标商品 1 个，match_type=direct
6. 受约束回复，总耗时 ~18s

### 场景 2：无匹配商品（"猫粮有没有"）

1. 向量检索无相关片段（全被 Rerank 丢弃）
2. SearchPlan：target=猫粮
3. 工具调用多轮均命中 0 条
4. 目标商品为空
5. LLM 自由回复告知无匹配，总耗时 ~11s

### 场景 3：跨域兜底（"想要游戏本打游戏"）

1. 向量检索命中轻薄本片段，Rerank 判定"非游戏本"全部丢弃
2. SearchPlan：target=游戏本，direct_terms=[游戏本]，fallback=[笔记本电脑]，allowed=[数码电子]
3. 工具调用命中笔记本电脑类商品（全部 match_type=fallback）
4. 目标商品 5 个，均为兜底替代
5. 最终回复基于兜底商品生成，总耗时 ~17s
## parallel_bootstrap 阶段说明

`parallel_bootstrap` 是对话工作流的启动阶段，主要职责不是做最终结论，而是尽快把可以并行启动的前置任务拉起来，为后续的工具循环、候选收敛和最终回复争取时间。

### 这一阶段做什么

- 并行准备用户意图上下文、历史摘要和基础状态占位。
- 尽早发起商品搜索计划和首轮工具规划；RAG 候选收集、直查候选收集由后续阶段接力完成。
- 为后续阶段准备一个稳定的事件结构，避免前端在首屏等待时遇到缺字段或状态跳变。

### 这一阶段不做什么

- 不负责最终商品推荐决策。
- 不负责把 SearchPlan、RAG 候选和工具结果完全收敛成最终白名单。
- 不负责替代后续的严格数据库校验和最终回复生成。

### 设计原则

1. 先点火，再收敛。
2. 只做轻量汇总，不做最终裁决。
3. 保持事件结构稳定，优先兼容前端消费。
4. 复用前置阶段已有结果，避免重复计算。

### 与后续阶段的关系

- `parallel_bootstrap` 负责把链路从“等待启动”推进到“已有上下文可用”。
- 后续工具循环和最终回复阶段会复用这里的上下文继续收敛。
- 如果前置候选已经足够明确，后续阶段可以更快进入整理结果，而不是继续扩展搜索范围。
