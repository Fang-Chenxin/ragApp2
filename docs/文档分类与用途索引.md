# 文档分类与用途索引

这个索引用来把当前 `docs/` 目录里的文档按用途分组，方便快速查找。  
如果你刚进入项目，建议先看“推荐阅读顺序”。

如果你想看最简入口，可以直接打开 [README.md](./README.md)。

## 1. 分类原则

- `docs/变更摘要/`：只放单次功能新增、优化、修复的变更记录
- `docs/项目验收/`：只放课程验收、答辩陈述、交付材料和代码规范对照材料
- `docs/` 根目录：放长期稳定的项目说明、架构说明、规范文档和使用指南
- 需要排查后端代码时：先看文件定位说明，再顺着服务层和子包进入

## 3. 变更摘要子目录用途

`docs/变更摘要/` 下每个文件对应一个功能主题或一次较完整的改动。  
命名规则通常是“功能特性描述”，例如：

- 导购流程并行化与目标商品白名单
- 模型选择与自定义模型管理
- 查询工具接入后端

建议在这里放：
- 新增功能说明
- 优化/修复说明
- 架构演进说明

不建议在这里放：
- 项目总览
- 验收汇报稿
- 使用教程
- 代码审查记录

## 5. 新文档放置建议

如果后续新增文档，可以按下面的规则放：

- 项目概览、架构、流程 -> `docs/`
- 验收材料、答辩稿、交付清单、代码规范对照 -> `docs/项目验收/`
- 单次功能变更、修复、优化 -> `docs/变更摘要/`
- 代码定位、调试方法、工程说明 -> `docs/`
- 使用说明、环境说明、端侧操作指引 -> `docs/`

如果某份文档长期只用于某一类人群，也可以考虑后续再单独建子目录；当前已经先把验收材料独立出来，避免根目录混杂。
