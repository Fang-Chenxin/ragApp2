## 视频目标

录制一个 5-10 分钟的带讲解演示视频，让评审快速理解本项目的完整形态：这是一个 **Android 原生客户端 + FastAPI 后端 + RAG 知识库 + SQLite 商品库 + LLM 工具调用** 的智能电商导购系统。

视频建议控制在 9-10 分钟。结构上先按 README 顺序从上到下介绍项目定位、总体技术、创新点、对话流程、并行处理机制、语义表策略、搜索与召回策略、核心能力、文件架构和启动配置，再切到 Android App 展示模型切换、多会话、不同导购模式，最后补充异常处理和测试验证。

## 开头总览

### 0:00-0:50 项目定位、技术栈与创新点

画面：

- 展示 Android App 首页。
- 切到根目录 `README.md`，展示标题、项目简介、技术架构表格和“项目亮点 / 创新点”。

讲解词：

> 大家好，这个作品是一个智能电商导购对话系统。它是 Android 原生客户端加 FastAPI 后端，不是 Web 套壳。后端结合 RAG 知识库、SQLite 商品库和 LLM 工具调用，前端负责流式聊天、模型选择、多会话历史、Markdown 和商品卡片展示。

> 用户可以像真实购物一样连续提问，系统会自动识别他是在随便看看、明确想买、追问更便宜、排除某个品牌或成分，还是需要多商品对比决策。

> 这个项目的亮点主要有三点：第一，最终导购不是让大模型自由发挥，而是通过 RAG 和 SQLite 商品库生成目标商品白名单，降低商品幻觉；第二，SearchPlan 会把真实购物语言解析成追问、排除、价格偏好和对比决策等结构化约束；第三，复杂对比在 Android 端会转成推荐排序和商品卡片矩阵，更适合手机阅读。

重点：

- Android 原生应用。
- FastAPI 后端。
- Chroma 向量库 + SQLite 商品库。
- 火山方舟 / OpenAI-compatible LLM。
- RAG + 工具调用 + 流式输出。

### 0:50-1:35 对话流程与并行处理机制

画面：

- 顺着 README 往下展示“对话处理流程”和“并行处理机制”。

讲解词：

> 一轮导购对话大致分为六步：第一步是需求分析，生成可折叠的“已思考”内容；第二步是 RAG 向量检索和 LLM 核验；第三步是 SearchPlan，把用户问题解析成结构化的目标商品、类目、排除条件、价格偏好和对比意图；第四步是 SQLite 商品库直查和工具调用；第五步是合并目标商品白名单；第六步是最终流式回复，并把历史、思考内容和商品清单保存下来。

> 这里最重要的是，最终回复不会随便编商品，而是被目标商品白名单约束。如果模型直接回复里提到了非白名单商品，后端会强制重写成受约束的最终回复。

> 这个流程也不是完全串行等待。后端在 `parallel_bootstrap` 阶段会尽早并行启动需求分析、SearchPlan 结构化规划和首轮工具规划；RAG 向量检索和 LLM 核验在后续 RAG 阶段执行，等待期间前端可以先收到需求分析增量；如果同一轮有多个 `query_products` 工具调用，也会并发执行，调试事件按完成顺序输出，回填给 LLM 的工具消息按原调用顺序排列。

> 所以日志里的各阶段耗时不能简单相加等于总耗时，因为分析、RAG、工具规划和部分商品查询存在重叠。后端会记录 `parallel_enabled` 和 `parallel_overlap_saved_estimate` 等字段，客户端断开时也会取消未完成后台任务。

流程口径：

```text
用户输入
→ 需求分析
→ RAG 检索与核验
→ SearchPlan 结构化规划
→ 语义表修正
→ SQLite 直查 / query_products 工具
→ 目标商品白名单
→ 受约束最终回复
→ Android 流式展示与历史保存
```

可强调字段：

- `parallel_enabled`
- `parallel_overlap_saved_estimate`
- `vector_search`
- `rag_rerank`
- `llm_calls`
- `tool_calls`
- `total`

### 1:35-2:20 语义表、同义词与搜索策略

画面：

- 继续展示 README 的“语义表与同义词策略”和“搜索与召回策略”。

讲解词：

> 这里补充一个很关键的业务知识层。项目没有把同义词和业务规则散落在代码里，而是集中放在 `search_semantics` 和 `attribute_ontology.json`。整体策略是：LLM 负责理解用户自然语言，语义表负责提供稳定的业务边界，代码负责执行和校验。

> `product_concepts` 管商品概念同义词，比如防晒霜、防晒乳、防晒；`category_aliases` 把用户口语词映射到数据库真实类目；`fallback_relations` 定义没有直接商品时哪些替代品可以推荐、哪些类目必须禁止；`brand_model_aliases` 处理 iPad、MacBook、Nike 这类品牌型号约束；`scenario_tags` 把学习、跑步、敏感肌、早餐等场景转成检索和排序提示。

> 另外 `attribute_ontology.json` 管品牌别名和属性标准化，比如颜色、容量、尺码、存储、口味、包装等。SearchPlan 阶段会把这些语义表命中结果注入 prompt，后处理阶段再做确定性修正，所以系统不是完全靠模型猜数据库字段，而是有一层可维护、可测试的业务知识表。

> 搜索上，这里不是只有一种检索。RAG 向量检索负责找商品知识片段，补充卖点和场景解释；SearchPlan 负责把用户口语变成结构化约束；SQLite 直查负责从真实商品库快速召回候选；`query_products` 工具让模型按结构化参数补充查询；fallback 替代搜索用于没有直接命中时找相邻商品；最后再通过白名单回查确保推荐商品真实存在。

可强调文件：

- `product_concepts.json`
- `category_aliases.json`
- `fallback_relations.json`
- `brand_model_aliases.json`
- `scenario_tags.json`
- `regression_cases.json`
- `attribute_ontology.json`

## 对话模式演示

本项目没有单独的模式开关，对话模式由 SearchPlan 根据用户自然语言自动识别。下面这些场景参考了 `test/test_search_plan_regression.py` 和 `test/test_history_and_stream_safety.py` 中的回归用例。

| 模式 | 推荐演示问题 | 预期展示 |
| --- | --- | --- |
| 探索浏览 | `随便看看有没有适合跑步的耳机` | `purchase_intent=browsing`，轻推荐、不催买 |
| 明确购买 | `推荐一款防晒霜` / `我想买个跑步耳机` | `purchase_intent=purchase_ready`，给出优先级和购买建议 |
| 追问承接 | `再便宜点的呢？` | `is_followup=true`，承接上一轮商品品类 |
| 价格偏好 | `再便宜点的呢？` / `方便面买哪一种更划算` | `price_preference=cheaper` |
| 品牌排除 | `除了耐克还有什么跑步鞋` | `excluded_brands` 包含 Nike/耐克 |
| 成分排除 | `不要含酒精的防晒` | `excluded_terms` 包含酒精 |
| 指定维度对比 | `对比这两款防晒，重点看价格和成分` | `comparison_dimensions` 包含价格、成分 |
| 性价比对比 | `方便面买哪一种更划算` | 识别对比意图和低价偏好 |
| 类目/子类目纠错 | `华为生态设备` | 非法或泛化子类目会被清理或兜底 |
| 替代推荐 | `肉松面包` | 没有直接商品时转向相邻品类/替代商品 |

## 异常与错误处理演示

### 8:15-9:00 连接失败与配置错误

建议展示 2-3 个，不需要全部录。

#### 场景 A：后端地址错误

操作：

1. 打开 Android 设置。
2. 把后端地址临时改为错误端口，例如：

```text
http://10.0.2.2:9999
```

3. 回到聊天页，刷新模型或打开对话列表。
4. 展示模型加载失败、向量模型状态不可达、会话列表加载失败等提示。
5. 立刻恢复正确地址。

讲解词：

> 这里演示后端不可达。客户端不会崩溃，而是用 Toast 或状态文本提示加载失败。恢复正确地址后，请求会重新正常工作。

#### 场景 B：后端地址格式校验

操作：

在后端地址设置里输入：

```text
10.0.2.2:8000
```

或：

```text
abc
```

讲解词：

#### 场景 C：自定义模型配置错误

#### 场景 D：工具参数校验错误
