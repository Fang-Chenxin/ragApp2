# 验收演示视频脚本

## 视频目标

录制一个 5-10 分钟的带讲解演示视频，让评审快速理解本项目的完整形态：这是一个 **Android 原生客户端 + FastAPI 后端 + RAG 知识库 + SQLite 商品库 + LLM 工具调用** 的智能电商导购系统。

视频建议控制在 9-10 分钟。结构上先按 README 顺序从上到下介绍项目定位、总体技术、创新点、对话流程、并行处理机制、语义表策略、搜索与召回策略、核心能力、文件架构和启动配置，再切到 Android App 展示模型切换、多会话、不同导购模式，最后补充异常处理和测试验证。

## 开头总览

### 0:00-0:50 项目定位、技术栈与创新点

画面：

- 展示 Android App 首页。
- 切到根目录 `README.md`，展示标题、项目简介、技术架构表格和“项目亮点 / 创新点”。

讲解词：

> 这是一个 Android 原生 + FastAPI 后端的智能电商导购系统。后端结合 Chroma 向量库、SQLite 商品库和 LLM 工具调用，前端负责流式聊天、模型选择、多会话和商品卡片展示。亮点有三：目标商品白名单降低幻觉；SearchPlan 把购物语言解析成结构化约束；语义表（search_semantics + attribute_ontology）把业务知识集中管理，避免规则散落在代码中。

### 0:50-1:35 对话流程与并行处理机制

画面：

- 顺着 README 往下展示“对话处理流程”和“并行处理机制”。

讲解词：

> 导购对话主要阶段包括：并行启动、RAG 处理、商品召回、约束过滤和最终回复。最终回复受目标商品白名单约束，非白名单商品会被强制重写。流程并非完全串行，后端采用流水线式并行策略：在 `parallel_bootstrap` 阶段同时启动需求分析、SearchPlan 和首轮工具规划，**启动后立即进入下一阶段，不等待完成**，随后立即启动 RAG 向量检索。因此需求分析、SearchPlan、首轮工具规划和 RAG 向量检索这四个任务几乎是同时并行执行的。在等待 RAG 处理完成期间，会穿插输出已生成的需求分析增量，避免前端空等。多个商品查询也会并发执行，因此日志各阶段耗时不能简单相加。

流程口径：

```text
用户输入
└─[并行启动] 需求分析 + SearchPlan + 首轮工具规划 + RAG向量检索（几乎同时启动）
    ├─ 需求分析增量穿插输出到前端
    ├─ 向量检索完成 → LLM rerank 核验 → RAG候选
    ├─ SearchPlan 完成 → SQLite 直查 → 直查候选
    ├─ 首轮工具规划完成 → 工具调用 → 工具候选
    ├─[合并去重] RAG候选 + 工具候选 + 直查候选 → 按优先级合并去重
    ├─[校验过滤] 数据库回查 + 约束过滤 → 目标商品白名单
    └─ 最终回复生成 + Android 展示 + 历史保存
```

可强调字段：

- `parallel_enabled`
- `vector_search`
- `rag_rerank`
- `llm_calls`
- `tool_calls`
- `total`

### 1:35-2:20 语义表、同义词与搜索策略

画面：

- 继续展示 README 的“语义表与同义词策略”和“搜索与召回策略”。

讲解词：

> 业务知识集中在 `search_semantics` 目录和 `attribute_ontology.json` 中统一管理，避免规则散落在代码里。语义表负责同义词映射、品类别名转换、替代推荐规则、品牌型号约束和场景标签，确保 LLM 的输出能准确匹配数据库真实值。
> 搜索策略采用多路召回：RAG 补充商品知识片段，SearchPlan 把口语转成结构化约束，SQLite 直查和 `query_products` 工具召回真实商品，fallback 处理无直接命中的情况，最后通过目标商品白名单确保推荐的商品真实存在。

可强调文件：

- `search_semantics/` 目录（包含 product_concepts、category_aliases、fallback_relations 等）
- `attribute_ontology.json`

## 对话模式演示

本项目没有单独的模式开关，对话模式由 SearchPlan 根据用户自然语言自动识别。下面这些场景参考了 `test/test_search_plan_regression.py` 和 `test/test_history_and_stream_safety.py` 中的回归用例。

| 模式 | 推荐演示问题 | 预期展示 |
| --- | --- | --- |
| 探索浏览 | `随便看看有没有适合跑步的耳机` | `purchase_intent=browsing`，轻推荐、不催买 |
| 明确购买 | `推荐一款防晒霜` / `我想买个跑步耳机` | `purchase_intent=purchase_ready`，给出优先级和购买建议 |
| 追问承接 | `再便宜点的呢？` | `is_followup=true`，承接上一轮商品品类 |
| 价格偏好 | `再便宜点的呢？` / `方便面买哪一种更划算` | `price_preference=cheaper` |
| 品牌排除 | `除了耐克还有什么跑步鞋` | `excluded_brands` 包含 Nike/耐克 |
| 成分排除 | `不要含酒精的防晒` | `excluded_terms` 包含酒精 |
| 指定维度对比 | `对比这两款防晒，重点看价格和成分` | `comparison_dimensions` 包含价格、成分 |
| 性价比对比 | `方便面买哪一种更划算` | 识别对比意图和低价偏好 |
| 类目/子类目纠错 | `华为生态设备` | 非法或泛化子类目会被清理或兜底 |
| 替代推荐 | `肉松面包` | 没有直接商品时转向相邻品类/替代商品 |

> 这是一个 Android 原生智能导购应用，后端是 FastAPI，结合 RAG、SQLite 商品库和 LLM 工具调用。项目通过 search_semantics 和 attribute_ontology 维护同义词、品类别名、fallback、品牌型号、场景标签和属性标准化，避免业务规则散落在代码里。导购链路采用流水线式并行策略：在 `parallel_bootstrap` 阶段同时启动需求分析、SearchPlan 和首轮工具规划，启动后立即进入下一阶段启动 RAG 向量检索，因此这四个任务几乎同时并行执行，等待期间穿插输出需求分析增量。RAG 检索负责补充商品知识，SQLite 直查和 `query_products` 工具负责召回真实商品，fallback 搜索负责无直接命中时寻找替代品，最后通过目标商品白名单避免编造商品；异常方面支持后端连接失败提示、配置校验、工具参数校验和流式异常历史保存。

## 录制注意事项

1. 开头技术介绍不要停留太久，控制在 2-3 分钟内。

6. 异常场景演示后要立刻恢复正确配置，避免影响后续录制。
