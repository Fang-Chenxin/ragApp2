## 4. 建议答辩口径

如果老师问“交付材料是否齐全”，可以直接这样回答：

> 我们已经准备了可运行 Demo、结构清晰的源代码仓库和配套技术文档。Demo 方面，Android 原生客户端与 FastAPI 后端可以端到端运行，支持流式对话和商品卡片展示；仓库方面，代码按 backend、android_app、docs、test 和数据集目录分层，便于检查；文档方面，已经补充了系统架构、处理流程、验收对照和 1 分钟答辩稿，便于演示和答辩。

如果老师问“代码规范是否达标”，可以这样回答：

> 仓库已经按职责分层，后端与客户端分目录管理；Python 依赖通过根目录唯一 requirements.txt 维护，Android 依赖通过 Gradle 维护；核心链路，尤其是 RAG、Prompt 构造和流式事件转换，都有明确注释，方便复现和检查。

## 5. 现阶段可改进项

下面这些点不影响当前交付，但如果要进一步提升规范性，可以继续补强：

| 项目 | 当前情况 | 建议 |
|---|---|---|
| 目录命名 | 不是字面上的 `/client /server` | 如果后续有重构计划，可以再做一层更强的命名统一；当前结构已经足够清晰 |
| 依赖版本锁定 | Python 依赖已通过根目录 requirements.txt 管理，但 Android 的依赖是 Gradle 管理 | 这属于 Android 标准做法，答辩时只需说明即可 |
| Demo 脚本化 | 当前可手动演示 | 如需更强展示性，可补充一份 demo 步骤清单或录屏脚本 |
