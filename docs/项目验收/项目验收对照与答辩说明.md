# 项目验收对照与答辩说明

## 1. 文档用途

本文档用于课程验收、答辩陈述和代码检查对照，目标是把本项目是否满足题目要求说明清楚：

- 从数据工程、模型编排、后端服务化到客户端体验，逐条核对实现情况
- 给出每条要求的对应代码证据，便于检查

> 说明：本文档中的源码引用均采用相对路径，便于在仓库内直接跳转查看。

---

## 2. 一句话总评

本项目已经实现了“Android 原生聊天端 + SSE 流式对话 + 向量检索 + SQLite 商品库校验 + 商品卡片展示”的完整闭环，能够处理模糊购物需求、检索库内商品、生成推荐理由并在客户端流式呈现。

其中最关键的工程约束是：

- 推荐结果不直接相信 LLM，而是先经过 RAG、SearchPlan、SQLite 白名单核验
- 最终输出只允许围绕已核验商品生成，尽量避免幻觉
- Android 端不是 Web 套壳，而是 Kotlin 原生实现流式渲染和商品卡片点击

## 5. 关键实现说明

### 5.1 为什么说“RAG 可靠性”比较强

这个项目不是“向量库查到什么就直接说什么”，而是做了多道约束：

1. 先从 Chroma 做语义召回
2. 再做 LLM rerank 核验
3. 再用 SearchPlan 约束查询范围
4. 再通过 SQLite 工具查商品库
5. 最终把商品回查成“白名单”
6. 最后回复只允许围绕白名单商品生成

对应代码见：

- [backend/service/tool_chat_service.py](../../backend/service/tool_chat_service.py)
- [backend/service/tool_chat/stream_stages.py](../../backend/service/tool_chat/stream_stages.py)
- [backend/service/tool_chat/stream_tool_loop.py](../../backend/service/tool_chat/stream_tool_loop.py)
- [backend/service/tool_chat/stream_final.py](../../backend/service/tool_chat/stream_final.py)

### 5.2 为什么说“商品事实可控”

商品相关事实不是让模型自由发挥，而是统一回到数据库：

- `product_id` 先做精确回查
- `base_price` 决定默认 SKU
- `image_path / landing_url` 从数据库字段派生
- `match_type` 区分 direct 和 fallback，避免把替代品说成点名商品

对应代码见：

- [backend/service/product_search/sqlite_search.py](../../backend/service/product_search/sqlite_search.py)
- [backend/service/tool_chat/product_selection.py](../../backend/service/tool_chat/product_selection.py)

### 5.3 为什么说 Android 端是“原生流式体验”

Android 端不是简单展示最终文本，而是：

- 先插入一个 assistant 占位消息
- 持续接收 SSE 事件
- 分别更新 status、analysis、content、selected_products
- 使用 RecyclerView 和 Adapter 局部刷新
- 商品卡片独立渲染，并支持点击跳转

对应代码见：

- [android_app/app/src/main/java/com/example/agentchat/MainActivity.kt](../../android_app/app/src/main/java/com/example/agentchat/MainActivity.kt)
- [android_app/app/src/main/java/com/example/agentchat/ChatAdapter.kt](../../android_app/app/src/main/java/com/example/agentchat/ChatAdapter.kt)

## 6. 目前的不足与后续扩展

### 6.1 已知不足

| 不足点 | 现状 | 影响 |
|---|---|---|
| 实时库存治理 | 仓库里没有独立库存服务或库存刷新闭环 | 价格和商品存在性较稳，但“库存实时生效”还不够完整 |
| 多模态采集 | 只有文本输入，没有图片/语音采集入口 | 课程里的多模态端侧能力尚未实现 |
| 显式意图分类器 | 主要靠 prompt 和 SearchPlan 做隐式分流 | 能用，但解释性不如独立分类器 |

### 6.2 后续扩展方向

- 增加库存服务或库存字段实时同步
- 增加图片上传 / 语音录入 / OCR 预处理
- 增加显式 intent classifier，区分“随便看看”和“明确购买倾向”

---

## 7. 结论

如果按课程的“最小功能交付标准”验收，本项目已经达标。  
如果按“四大维度完整覆盖”验收，当前主要缺口是：

- 实时库存治理
- 图片 / 语音多模态采集

但从整体工程链路看，项目已经完成了从“模糊需求理解”到“商品检索、推荐生成、流式展示”的闭环，且有较强的防幻觉与数据一致性意识，适合作为答辩展示版本。
