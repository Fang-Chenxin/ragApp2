# 变更摘要文档撰写规范

## 0. 如何调用本规范

### 0.1 首次调用（任务开始前）
**当用户要求进行功能开发/修改时，在开始工作前必须执行：**

1. 主动查看本规范文档：[变更摘要文档撰写规范.md](file:///home/fang/Documents/trae_projects/ragApp/docs/变更摘要文档撰写规范.md)
2. 查看现有变更摘要文档，了解项目文档风格
3. 根据任务复杂度选择文档版本（完整版/简洁版）

### 0.2 任务进行中
- 记录新增/修改的文件清单
- 收集技术要点和变更说明

### 0.3 任务完成后（必须执行）
**任务完成后必须按顺序完成：**

```
1. ✅ 创建/更新详细变更摘要文档（docs/变更摘要/）
2. ✅ 更新 CHANGELOG.md（根目录）
3. ✅ 更新 项目介绍说明.md（docs/项目介绍说明.md）
4. ✅ 按需更新 README.md（根目录）
5. ✅ 对照「撰写检查清单」逐一验证
```

**说明**:
- `CHANGELOG.md`: 必须更新，记录每个版本的新增、修改、修复内容
- `项目介绍说明.md`: 必须更新，保持项目整体介绍与最新功能同步
- `README.md`: 按需更新，仅在新增重要功能或修改用户使用流程时更新
- `示例文档索引`（第6章）：**不需要更新**，仅作为编写参考模板使用

---

## 0.5 标准工作流程（必须遵守）

### 完整开发流程
```
用户提出需求 → 
├─ 1. 查看本规范文档
├─ 2. 分析需求，制定计划
├─ 3. 开发功能
├─ 4. 测试验证
├─ 5. 创建变更摘要文档（docs/变更摘要/）
├─ 6. 更新 CHANGELOG.md
├─ 7. 更新 项目介绍说明.md（docs/项目介绍说明.md）
├─ 8. 按需更新 README.md
└─ 9. 对照检查清单验证完成
```

### 确保规范执行的方法
1. **任务开始前**: 主动声明「我会先查看项目变更文档规范」
2. **任务进行中**: 主动记录文件变更和技术要点
3. **任务结束前**: 先完成文档工作，再告知用户「任务已完成」
4. **检查清单**: 最后一步必须对照「第9章 撰写检查清单」逐一验证

## 3. 格式规范

### 3.1 图标使用

| 图标 | 含义      | 使用场景       |
| -- | ------- | ---------- |
| 🔴 | 高       | 影响范围大、严重风险 |
| 🟡 | 中       | 中等影响、普通风险  |
| 🟢 | 低       | 影响小、低风险    |
| ✅  | 已完成/已修复 | 功能完成、问题修复  |
| ✨  | 亮点/新特性  | 新功能、技术亮点   |
| 📱 | 移动端     | Android相关  |
| 🔧 | 工具/配置   | 配置管理、工具类   |
| ⚠️ | 警告      | 破坏性变更、需要注意 |

### 3.2 代码块规范

- **Kotlin**: 使用 \`\`\`\`kotlin\` 标记
- **Python**: 使用 \`\`\`\`python\` 标记
- **XML**: 使用 \`\`\`\`xml\` 标记
- **Shell/Bash**: 使用 \`\`\`\`bash\` 标记
- **通用代码**: 使用 \`\`\`\`text\` 标记

### 3.3 表格规范

- 表格标题简洁明了
- 列数控制在合理范围（建议≤6列）
- 对齐方式：标题居中，内容左对齐

### 3.4 代码引用规范

- **文件引用**: 使用相对路径，格式 `[文件名](file:///绝对路径)`
- **行号引用**: 格式 `[文件名](file:///绝对路径#L起始行-L结束行)`
- **方法引用**: 格式 `[方法名](file:///绝对路径#L行号)`
