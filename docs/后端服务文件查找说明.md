# 后端服务文件查找说明

## 适用范围

本文用于快速定位并理解后端服务代码。后端目录按“应用入口 + API 路由 + 服务对象 + 领域子包”组织：

- `backend/main.py` 负责创建 FastAPI 应用、注册路由、管理启动/关闭生命周期。
- `backend/api/` 负责 HTTP/SSE 协议层，只做参数校验、响应包装和服务调用。
- `backend/service/` 负责业务服务、模型调用、RAG、历史记录、导购工具编排。
- `backend/service/product_search/` 是商品库检索引擎和 OpenAI 工具定义。
- `backend/service/tool_chat/` 是导购聊天的阶段化流程，尤其是流式 SSE 的内部实现。
- `backend/config/` 负责环境变量、模型列表、日志格式等运行配置。

阅读代码时建议先确认“请求从哪个 API 进入”，再顺着服务对象进入对应子包，不要从 mixin 文件随机开始看。

## 应用入口与配置

| 文件 | 具体功能 | 修改时关注点 |
|------|----------|--------------|
| `backend/main.py` | 创建 FastAPI 应用；配置 CORS；注册 `chat`、`knowledge`、`product-search` 路由；在 `lifespan()` 中初始化和清理所有服务；提供 `/` 和 `/health`。 | 新增路由要在这里 `include_router`；启动期依赖异常会导致服务整体失败；`/health` 目前只粗略检查 LLM 和向量库。 |
| `backend/config/settings.py` | 用 Pydantic Settings 读取 `.env` 和环境变量；管理 LLM 默认模型、可选模型、自动发现模型来源、Embedding/Chroma/SQLite 路径、RAG 参数、日志参数。 | 模型列表的 `api_key_env` 会从真实环境或 `backend/.env` 取值；`available_llm_models` 为空时才把 `llm_model` 加入列表；RAG 超时、rerank 开关、并行开关都在这里。 |
| `backend/config/logging_config.py` | 初始化根 logger；控制台只显示关键日志，文件记录 DEBUG 级完整链路；过滤 `service.tool_chat` 等高频处理日志。 | 如果排查导购链路细节，优先看 `backend.log`；不要随意放开控制台 DEBUG，否则 SSE 请求会刷屏。 |
| `backend/config/__init__.py` | 导出 `settings` 和 `Settings`。 | 配置模块公共入口。 |
| `backend/__init__.py` | 标记 `backend` 为 Python 包。 | 通常无需修改。 |

## API 路由层

| 文件 | 具体功能 | 主要入口 |
|------|----------|----------|
| `backend/api/chat.py` | 聊天与会话 API。`/api/models` 返回可用模型；`/api/chat` 和 `/api/chat/stream` 返回 SSE 流式导购结果；会话 CRUD 与历史读取/清空也在这里。 | `chat_endpoint()` 是聊天主入口；内部把 `tool_chat_service.chat_with_tools_stream()` 的事件转换成前端 SSE JSON；`save_history_once()` 负责最终保存用户消息、助手回复和需求分析。 |
| `backend/api/knowledge.py` | 知识库管理 API。支持添加文档到 Chroma，读取知识库文档数量。 | `add_knowledge()`、`get_knowledge_stats()`。 |
| `backend/api/sqlite_product_search.py` | 商品搜索调试/工具 API。支持自然语言搜索、结构化搜索、工具 schema 查看、工具执行、健康检查。 | `/search/text` 走规则解析；`/search` 走结构化过滤；`/tool/spec` 和 `/tool/run` 供调试 Function Calling。 |
| `backend/api/__init__.py` | API 路由导出索引。 | 当前只导出 `chat_router`、`knowledge_router`，商品搜索路由在 `main.py` 直接导入。 |

## 顶层服务入口

| 文件 | 何时查看 |
|------|----------|
| `backend/service/__init__.py` | 查看服务初始化顺序、全局服务对象导出、启动/清理流程。`initialize_services()` 会按 LLM -> Embedding -> Chroma -> RAG -> SQLite -> ToolChat 的顺序装配依赖。 |
| `backend/service/llm_service.py` | 查看 LLM 客户端、模型请求、流式/非流式调用封装。支持服务端默认模型、服务端发现模型、客户端本地模型配置三类调用。 |
| `backend/service/rag_service.py` | 查看通用 RAG 服务、Embedding、Chroma 向量库读写。`VectorStore.query_with_sources()` 会根据问题推断品类过滤，并在过滤无结果时回退无过滤检索。 |
| `backend/service/history_service.py` | 查看对话历史、多会话、文件锁、原子写入逻辑。历史按用户 hash 分目录，每个会话一个 JSON，元数据在 `meta.json`。 |
| `backend/service/tool_chat_service.py` | 查看导购工具聊天的对外入口。类本身由多个 mixin 组成；非流式 `chat_with_tools()` 保留完整流程，流式逻辑委托给 `tool_chat/` 子包。 |

## 工具聊天子包

目录：`backend/service/tool_chat/`

| 文件 | 何时查看 |
|------|----------|
| `__init__.py` | 查看工具聊天内部模块索引 |
| `stream_pipeline.py` | `chat_with_tools_stream()` 的入口。创建 `_StreamPipelineContext`，按“并行启动 -> RAG -> 分析收尾 -> 商品直查 -> 工具循环 -> 最终回复”执行阶段，并在 finally 中取消后台任务。 |
| `stream_context.py` | 流式共享上下文。`_StreamTaskGroup` 记录后台任务；`_StreamPipelineContext` 保存一轮 SSE 请求中所有跨阶段状态，如 timings、分析文本、RAG 来源、工具结果、目标商品。 |
| `stream_stages.py` | 基础阶段。负责启动需求分析流、后台商品直查、首轮工具规划；执行向量检索和 LLM rerank；把分析队列转换成前端 `analysis` 事件。 |
| `stream_tool_loop.py` | 工具循环。每轮先让 LLM 规划工具调用，再并发执行 SQLite 查询工具，按工具调用顺序回填 `tool` 消息；连续空参数会触发断路器；找到足够候选会提前进入最终回复。 |
| `stream_final.py` | 最终阶段。合并直查商品和工具商品，输出 `selected_products` 调试事件；构造受目标商品约束的最终推荐 prompt；流式输出 `content`，最后输出 `done`。 |
| `rag.py` | 工具聊天链路中的 RAG 辅助。处理向量检索超时、构建 rerank prompt、解析 LLM JSON 评分、按阈值保留/丢弃知识片段。 |
| `prompts.py` | Prompt 构造。包含需求分析 prompt、工具规划 prompt、最终系统 prompt，以及从历史回复中提取商品 ID/品牌/名称作为下一轮检索上下文。 |
| `product_selection.py` | 目标商品选择。提取工具结果中的商品、去重合并、按 ID 回查数据库校验、按用户显式品类过滤，构造最终推荐消息并清洗对外回复中的内部 ID。 |
| `trace.py` | SSE 调试事件和日志格式化。统一生成 `status`/`debug` payload，提取 RAG 来源，格式化 `backend.log` 中的向量分数、rerank 明细、工具结果、目标商品。 |

常见定位：

| 问题 | 优先查看 |
|------|----------|
| 流式接口不返回 `done` | `tool_chat/stream_pipeline.py`、`tool_chat/stream_final.py` |
| 需求分析内容异常 | `tool_chat/stream_stages.py`、`tool_chat/prompts.py` |
| 工具调用参数为空或循环过多 | `tool_chat/stream_tool_loop.py`、`tool_chat/prompts.py` |
| 最终推荐商品不符合用户品类 | `tool_chat/product_selection.py` |
| RAG 候选被过滤或排序异常 | `tool_chat/rag.py`、`tool_chat/trace.py` |
| 前端需要排查 SSE 调试事件 | `tool_chat/trace.py`、`tool_chat/stream_stages.py`、`tool_chat/stream_tool_loop.py` |

## 商品检索子包

目录：`backend/service/product_search/`

| 文件 | 何时查看 |
|------|----------|
| `__init__.py` | 查看商品检索内部模块索引 |
| `engine.py` | 商品搜索核心引擎。读取 `attribute_ontology.json`；解析自然语言中的品类/品牌/属性；扩展同义词、颜色、容量、尺码等变体；生成 SQL；执行 FTS 查询并在无结果时用 LIKE 回退。 |
| `sqlite_search.py` | 服务对象封装。检查数据库是否存在；对外提供自然语言搜索、结构化搜索、按 product_id 精确回查；数据库不可用时返回空的模拟结果。 |
| `query_tool.py` | OpenAI Function Calling 工具适配层。定义 `query_products` schema；执行工具调用；优先处理 `text` 自然语言查询，若没有 `text` 则校验结构化参数。 |

常见定位：

| 问题 | 优先查看 |
|------|----------|
| 自然语言商品搜索解析不准 | `product_search/engine.py` |
| 商品库不可用或路径异常 | `product_search/sqlite_search.py` |
| 工具 schema、参数校验、空参数报错 | `product_search/query_tool.py` |
| 商品 ID 精确回查不稳定 | `product_search/sqlite_search.py` |

## API 与服务的关系

| API 文件 | 主要依赖 |
|----------|----------|
| `backend/api/chat.py` | `service.tool_chat_service`、`service.history_service`、`service.llm_service`、`config.settings` |
| `backend/api/sqlite_product_search.py` | `service.sqlite_product_search_service`、`service.product_search.query_tool` |
| `backend/api/knowledge.py` | `service.vector_store`、`config.settings` |

`service/__init__.py` 会继续导出部分历史公共名称，例如 `sqlite_product_search_service` 和 `sqlite_product_query_tool`，用于保持 API 层和旧测试的调用方式稳定。新代码若直接访问内部实现，优先使用 `service.product_search.*` 或 `service.tool_chat.*`。

## 关键调用链

### 流式导购聊天

1. 前端请求 `/api/chat` 或 `/api/chat/stream`。
2. `api/chat.py::chat_endpoint()` 确认会话、模型配置、LLM 可用性，并构造 SSE 生成器。
3. `tool_chat_service.chat_with_tools_stream()` 创建上下文并执行阶段列表。
4. `stream_stages.py` 同时推进需求分析、RAG 检索、商品直查、首轮工具规划。
5. `stream_tool_loop.py` 执行 LLM 工具规划和 SQLite 工具查询。
6. `product_selection.py` 合并并校验目标商品。
7. `stream_final.py` 生成最终回复，输出 `done`。
8. `api/chat.py` 收到 `done` 后保存历史。

### 商品查询工具

1. LLM 生成 `query_products` 工具调用。
2. `tool_chat_service._run_tool_worker()` 在线程中调用 `product_search.query_tool.run_tool()`。
3. `query_tool.run_tool()` 根据参数选择自然语言查询或结构化查询。
4. `sqlite_search.SQLiteProductSearchService` 调用 `engine.agent_search_by_rule_parsed_text()` 或 `engine.agent_search_products()`。
5. `engine.py` 解析 ontology、构造 SQL、执行 SQLite 查询并返回商品列表。

### 知识库 RAG

1. `VectorStore.query_with_sources()` 向 Chroma 查询相似片段。
2. `ToolChatRagMixin._rerank_context_docs_with_llm()` 可选调用 LLM 给候选片段评分。
3. `ToolChatTraceMixin` 把 RAG 来源和检查结果转成 debug 事件与日志。
4. 最终系统 prompt 将保留的知识片段作为背景，但商品事实仍以 SQLite 工具结果为准。

## 修改建议

1. 先从顶层服务入口确认调用链，再进入对应子包修改细节。
2. 修改流式导购链路时，优先确认是否影响 SSE 事件协议。
3. 修改商品搜索逻辑时，区分“查询解析/SQL 构建”与“工具 schema/执行分发”。
4. 新增工具聊天内部能力时，优先放入 `tool_chat/` 子包；新增商品检索内部能力时，优先放入 `product_search/` 子包。
5. 若新增长期模块，请同步更新本文和对应子包的 `__init__.py` 阅读索引。
6. 修改历史服务时注意文件锁和原子写入，避免并发请求写坏 JSON。
7. 修改模型配置时同时检查 `/api/models` 返回、`llm_service._resolve_client()` 调用路径和客户端本地模型配置。
8. 修改 RAG 参数时同步观察 `backend.log` 中 `vector_search`、`rag_rerank`、`rag_sources` 事件，确认不是因阈值或超时导致上下文被丢弃。
