# RAG 电商导购 Agent 系统架构与处理流程总结

## 一、系统整体架构

### 1.1 技术栈
| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **移动端** | Kotlin 原生 Android | 非 Web 套壳，完全原生体验 |
| **后端框架** | FastAPI (Python) | 异步高性能 Web 框架 |
| **向量数据库** | ChromaDB | 本地持久化，用于知识库检索 |
| **关系数据库** | SQLite | 结构化商品数据存储 |
| **大模型接入** | 火山方舟 / OpenAI-compatible | 多模型支持，流式响应 |
| **Embedding** | Chroma 默认本地 embedding / 火山方舟多模态 Embedding（可选） | 向量化知识片段 |

## 二、核心处理流程

### 2.1 请求处理主流程
当用户发送一条导购查询时，系统按以下流程处理：

```
用户查询

    ▼
```

### 2.2 详细流程说明

#### 阶段 1：并行启动（Parallel Bootstrap）
当配置 `tool_chat_parallel_enabled=True` 时，启动阶段会先并行拉起需求分析、SearchPlan 和首轮工具规划。随后 RAG 检索、SearchPlan 收尾和 SQLite 直查会按阶段继续执行，并复用前面已经启动的任务结果。

| 任务 | 说明 | 耗时参考 |
|------|------|---------|
| **需求分析** | LLM 分析用户意图，提取关键信息 | ~1-2s |
| **SearchPlan** | LLM 输出目标商品、直查词、兜底词、品类约束和排除条件 | ~1-2s |
| **首轮工具规划** | LLM 决定是否调用工具及参数；并行开关关闭时在工具循环内串行执行 | ~1-2s |
| **RAG 检索/核验** | ChromaDB 知识库语义检索并可选 LLM rerank | ~0.5-30s |
| **商品直查** | SearchPlan 可用后按原始需求直接查询 SQLite | ~0.1-0.3s |

#### 阶段 2：RAG 重排序
```
向量检索结果（候选片段）
        │
        ▼
┌─────────────────────────────────────┐
│  LLM 评分器                          │
│  - 输入：用户问题 + 候选片段          │
│  - 输出：每个片段的评分（1-5 分）      │
│  - 超时：默认 30 秒                  │
└─────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────┐
│  过滤逻辑                            │
│  - 评分 < 2 分 → 丢弃                │
│  - 仅 1 条候选 → 跳过检查直接使用      │
│  - 超时/错误 → 保留原始向量排序        │
└─────────────────────────────────────┘
        │
        ▼
  保留的知识片段（用于最终回复）
```

#### 阶段 3：工具调用循环
```python
# 工具调用流程伪代码
for round in range(max_tool_calls):  # 默认 5 轮
    # 1. LLM 决定是否调用工具
    response = llm.chat_with_tools(messages, tools)
    
    if response.has_tool_calls:
        # 2. 执行工具调用（可并行多个）
        for tool_call in response.tool_calls:
            result = run_tool(tool_call.name, tool_call.arguments)
            messages.append(tool_result)
        
        # 3. 检查是否已找到足够商品
        if len(selected_products) >= 3:
            break  # 提前退出
    else:
        # 4. LLM 返回文本，结束循环
        break
```

#### 阶段 4：商品查询工具
```python
# query_products 工具支持两种查询模式

# 模式 1：自然语言查询
{
    "text": "银色手机"
}
# → 规则解析 → 提取 keyword/brand/category/attr_filters → SQL

# 模式 2：结构化查询
{
    "keyword": "手机",
    "brand": "苹果",
    "category": "数码电子",
    "attr_filters": [{"key": "颜色", "value": "银色"}]
}
# → 直接构建 SQL
```

**规则解析流程**：
```
用户文本 "银色手机"
    │
    ▼
┌─────────────────────────────────────┐
│  1. 分词提取                         │
│  2. 品牌匹配（brands 映射表）         │
│  3. 分类匹配（category_aliases）      │
│  4. 属性族匹配（families 映射表）      │
│     - 颜色：银色 → 银灰/银白/...      │
│     - 存储：1TB → 1024GB/1024GB SSD  │
└─────────────────────────────────────┘
    │
    ▼
{
    "keyword": "手机",
    "brand": null,
    "category": null,
    "attr_filters": [{"key": "颜色", "value": "银灰"}]
}
```

## 五、SSE 事件类型

客户端通过 Server-Sent Events 接收流式响应：

```typescript
// 事件类型定义
interface SSEEvent {
  type: 'status' | 'analysis' | 'rag_sources' | 'selected_products' | 'content' | 'done' | 'error';
  
  // status: 进度状态
  phase?: 'need_analysis' | 'retrieving_knowledge' | 'reranking_knowledge' | 
          'querying_products' | 'organizing_results' | 'saving_history';
  
  // analysis: 需求分析内容
  analysis?: string;
  summary?: string;
  
  // rag_sources: 知识来源商品
  rag_sources?: Array<{
    product_id: string;
    title: string;
    brand: string;
    category: string;
  }>;
  
  // selected_products: 目标商品
  selected_product_ids?: string[];
  selected_products?: Array<{
    product_id: string;
    title: string;
    brand: string;
    base_price: number;
  }>;
  
  // content: 回复内容片段
  content?: string;
  
  // done: 完成信号
  history_saved?: boolean;
  timings?: {
    analysis_calls: number;
    vector_search: number;
    rag_rerank: number;
    llm_calls: number;
    llm_rounds: number;
    tool_calls: number;
    tool_rounds: number;
    total: number;
  };
  
  // error: 错误信息
  error?: string;
}
```

## 六、性能优化策略

### 6.1 并行执行
- 启动阶段并行执行需求分析、SearchPlan 和首轮工具规划；RAG 检索和 SQLite 直查在后续阶段接力执行并复用上下文
- 预估节省时间：~2-3 秒

### 6.2 提前退出
- 当找到 ≥3 个目标商品时，提前进入回复生成阶段
- 避免不必要的工具调用轮次

### 6.3 超时控制
- 向量检索超时：默认 3 秒
- LLM 重排序超时：默认 30 秒
- 超时后降级使用原始结果

### 6.4 断路器
- 连续 2 次空参数工具调用后，提前退出工具循环
- 避免无效的 LLM 调用

## 八、错误处理与降级策略

| 场景 | 降级策略 |
|------|---------|
| 向量检索超时 | 跳过知识库上下文，继续流程 |
| LLM 重排序超时 | 保留原始向量排序结果 |
| LLM 重排序失败 | 保留原始向量排序结果 |
| 工具调用失败 | 返回错误信息，LLM 尝试其他工具或直接回复 |
| LLM 调用失败 | 返回错误信息给客户端 |
| 目标商品为空 | 结合 RAG、需求分析和库存事实保守回复，不编造商品 |
| 受约束回复为空 | 使用确定性兜底回复 |

---

## 九、总结

本系统采用 **RAG + Tool Calling** 混合架构，核心特点：

1. **并行化处理**：需求分析、知识检索、工具规划并行执行，提升响应速度
2. **双通道检索**：向量检索（语义）+ SQLite 查询（结构化），互补覆盖
3. **LLM 重排序**：确保知识片段质量，减少幻觉
4. **受约束回复**：基于真实商品数据生成推荐，保证准确性
5. **流式响应**：实时推送进度和内容，提升用户体验
6. **容错降级**：超时/失败时自动降级，保证服务可用性
